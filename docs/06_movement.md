# 06 — Movement: Transactions, Proof Chains & UILP (Phase 6)

**What this phase built:** the *movement* layer — **transfer** a token between accounts, record it on a
(stand-in) **unified ledger** as a tamper-evident **state-commitment chain**, and attach a real **Merkle
proof of inclusion** that the browser re-verifies with only hashes. The headline demonstration: **a transfer
happens, its proof folds `leaf → root` to match the ledger's published root, and flipping one bit of the leaf
breaks it.** Teach-first, tied to the paper (§5.4.4 unified ledger, §5.4.6 UILP / proof chains) and the exact
schema fields. Bold terms are in [`00_glossary.md`](00_glossary.md). This reuses Phase 5's `state` /
`stateCommitment` fields directly.

## Why this layer exists (the problem)

A token that can't move is a fancy database row. But moving value *between institutions* is exactly where the
old system needs intermediaries and reconciliation — because there's no shared, trustable record of "what
happened." The paper's answer (§5.4.4, §5.4.6) is a **unified ledger** that records state changes plus a
**proof** of each one, so *any* other ledger can verify a transfer occurred **without trusting the ledger that
recorded it** — just by checking hashes. That's the foundation for value crossing ledgers.

## 1. Two levels of transaction
**Spec:** `schemas/transaction/{token-transaction,transaction-log}.schema.json`.
- A **`TokenTransaction`** is a *per-token* state change ("this token went A → B"), carrying `stateBefore`
  and `stateAfter` commitments. Required: `tokenTxId, txId, tokenId, operation, stateAfter, timestamp`.
- A **`TransactionLog`** is the *business-transaction envelope* wrapping one or more TokenTransactions, with
  `initiator`, `status`, `timestamps`, `proofProfile`, `ledgerAnchors`. Required: `txId, initiator,
  tokenTransactions, status, timestamps`.

Why two levels: one business action (a swap, a double-entry payment) can touch several tokens, and each token
needs its own auditable transition record. **Our code:** `standin-service/src/ledger.ts` builds both, and
`/v1/transaction/get` + `/v1/token/transactions` return them (validated against `token-interfaces.yaml`).

## 2. The state-commitment chain (tamper-evidence over history)
**Spec:** `StateReference.stateCommitment` — *"State commitments form a chain where each commitment includes
the previous one, enabling tamper detection."* A **state commitment** is `sha256(previousCommitment +
txId + tokenId + owner + timestamp)`. Because each folds in the previous, a token's history is a **hash
chain**: alter any past state and every later commitment diverges. This is the Phase-3 signature idea
(tamper-evidence of one message) applied to a whole *history*. **Our code:** `ledger.recordMint` sets the
genesis commitment; `ledger.recordTransfer` chains the next one (`stateAfter.previousCommitment` literally
points back). You can watch the token's `state.stateCommitment` change after a transfer.

## 3. The Merkle proof (the UILP centerpiece)
**Spec:** `ProofDetails` — `leafHash`, `merkleRoot`, `proofPath[{hash, direction}]`, `leafIndex`,
`proofStatus`. A **Merkle tree** hashes every transaction (a **leaf**) pairwise up to a single **`merkleRoot`**
that commits to all of them. To prove one transaction is included you ship only a **`proofPath`** — one
sibling hash per level — and the verifier re-folds `leafHash → root`. If it equals the (publicly anchored)
root, inclusion is proven **without holding the data**. That compact, independently-checkable proof is what
**UILP (Universal Ledger Interoperability Protocol)** carries between ledgers; the paper calls the growing
record of them a **proof chain** (§5.4.6). **Our code:** `ledger.getProof` builds the tree from all leaves and
returns the path; the **browser** re-verifies it in `app/src/lib/crypto.ts` → `foldMerkleProof`, needing only
SHA-256 — the same "verify it yourself, trust no one" story as Phases 3–4.

## 4. What UILP solves (even though our ledger is one process)
UILP is the protocol by which **ledger B accepts a proof generated by ledger A**. Ledger A anchors its Merkle
root somewhere both can see; A hands B a transaction + its proofPath; B folds to the root and — if it matches
the anchored root — accepts that the transaction happened, without replaying A's history or trusting A. Our
stand-in runs **one** ledger, so we don't perform cross-ledger settlement; but the proof it emits is exactly
the artifact UILP would carry, and the browser plays the role of "the other party verifying." (The
`ledgerAnchors` field on our TransactionLog is where a real anchor to a public chain would go — ours is a
fabricated placeholder, marked in code.)

## Audit
Administrative changes (not transfers) get their own record: **`AuditEvent`**
(`schemas/transaction/audit-event.schema.json`) — `MetadataUpdated`, `ClaimAttached`, `ClaimRevoked`, etc.,
with `before`/`after` change details and approvals. There's no audit *endpoint* in the vendored API, so we
don't build one this phase; the transfer trail above already gives an auditable, provable history via
`token/transactions` + `transaction/proof`.

## The stand-in, stated plainly

Finternet's real Unified Ledger provider (`solana-ul-provider`) does **not build** here (a bit-rotted pinned
dependency; established in Phase 1). So `ledger.ts` — the in-memory ledger + Merkle/proof engine — is ours
(`// STAND-IN:`; logged in [`01_reference_code_status.md`](01_reference_code_status.md)). **What is real:**
the SHA-256 state-commitment chain and the Merkle proofs, independently re-verified in the browser; and the
endpoint/transaction shapes, validated against the real `token-interfaces.yaml`. The `/v1/transaction/proof`
path isn't in the vendored spec (only its `ProofDetails` component is), so we **inject the obvious binding
in-memory** (`spec.ts` shim #4) to validate our proof responses against their real `ProofDetails` schema.
**What is stand-in:** one process instead of a distributed/anchored ledger, in-memory storage, and a
placeholder `ledgerAnchor` (no real public-chain anchoring).

## Verified end-to-end (Phase 6)

Over HTTP **and through the Vite proxy**:

- **Mint** sets an opening `stateCommitment`; `token/transactions` shows one `mint` record.
- **Transfer** (`token/transact`, `operation: transfer`) → `202 accepted`; the token's `owner` identity
  changes and its `stateCommitment` **advances** (the new record's `stateBefore` equals the old head).
- **`transaction/get`** → a `TransactionLog` (`status: completed`, `proofProfile: merkle`, a ledger anchor).
- **`transaction/proof`** → a `ProofDetails` (`proofStatus: proven`); re-folding `leafHash + proofPath`
  reproduces `merkleRoot` — and a tampered leaf does not.

Tests: `standin-service/src/ledger.test.ts` (4) — mint records genesis; transfer moves ownership + chains the
commitment; the Merkle proof folds leaf→root and tampering breaks it; non-owner transfer refused. All 16
service tests pass; app typecheck clean.
